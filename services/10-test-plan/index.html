Test Plan

A plan for evaluating the functionality of the prototype to verify that all performance objectives and specifications have been met.

Once the prototype is made, testing the functionality of the device can be implemented by a testing pass/fail list. This list not only has individual requirements, but compounded requirements where two functions might integrate. While it may be impossible to test all configurations of compounding functions, a good general umbrella test should also be implemented. Such as data transfer rate at specific battery levels, not all battery levels.

Functions to be tested with testing parameters:

Power on/off button

The power on/off button should be tested at different battery levels: fully charged, half way, depleted, and deep discharge of battery. The power on/off button should work in the fully charged and halfway mark. In the depleted mark there should be a red LED flashing indicator of low power. In the deep discharge this LED will not have the power to blink.

Menu Navigator buttons

These buttons are used in setup and connect mode. While performing setup and connect mode all software accessible modes are to be selected. This includes having multiple devices that use the same bluetooth connection methods in active mode in order to determine device choice selection.

JTAG Readout

JTAG is used only internally for debugging. When JTAG is connected, all software status should be readable and an automated check for all possible status should be collected.

Sensor data collection

Via JTAG a manually connection should be able to pull the most recent data. This is to be compared to the same data with the same time stamp via bluetooth connection. This should be done at 3 battery levels. Full, half way, and 10% usable battery remaining.

Battery Charger

The system should be tested against various commonly available charging devices. This includes, under voltage, over voltage, under current and over current. The device shall specify a set low current, with a voltage range. The device shall not be internally damageable via leaving the device on the charger.

Sensor sensitivity

The sensor works best while in direct contact with the skin. While movement occurs with the device, data can become discontinuous. The system shall ‘smooth’ this data so that discrete falls and jumps under 0.25 seconds by brief separation do not become ‘true’ data inputs. While this is a default state the device shall be in, it can be turned off for specific requirements by VR developers.

Thermal Heat

While the device is under various working loads. Thermal readings shall be taken and not surpassed by UL device specifications. This includes testing with a neighboring heat source at 98.6 F (human). As well as common temperature ranges between 40 - 115 F, and humidity conditions between 0 -100%. Benchmarked each 25 degree temperature change with a 20% humidity change. Note: this is 3 different temperature set points, each with 5 different humidity values.

VR connections

The device shall be tested connected with the top 5 VR headsets purchasable. While they all should use the same standard for bluetooth. These should be validated.

Breakaway clip

Because the device is secured around the neck, the clip has a max ‘breakaway’ force of 5 lbs. This should be tested and true for vertical and horizontal pull forces. The most common direction being horizontal with a set design value of 5 lbs and vertical being able to decrease to 3lbs, but not to increase by more than 6 lbs.

Cycling buttons

Buttons should be machine cycled to last for 10,000 cycles minimum. But testing until failure to create known data failure points. The same testing should be done in the various temperature and humidity conditions as the thermal heat testing.

Power cycle

The device shall be power cycled and designed to last a minimum of 5000 times. But should be done until failure. The same testing should be done in the various temperature and humidity conditions as the thermal heat testing.
